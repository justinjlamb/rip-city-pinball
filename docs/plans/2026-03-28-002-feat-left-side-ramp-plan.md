# Left-Side Ramp with Real Physics

## Overview

Add a ramp on the left side of the playfield. The ball enters from the lower-left going upward, travels up and across the upper playfield along a visible canvas-drawn path, and exits on the right side dropping back toward the flippers. Real physics — the ball rolls along guide walls with reduced gravity while on the ramp, and can fail to complete if too slow.

## Problem Frame

The upper playfield has bumpers and a scoop but no directional feature that rewards aim. A ramp gives the player a target to shoot for from the left flipper, creates a satisfying arc across the table, and awards points for completion. It's the last major gameplay element before the game is a complete MVP.

## Requirements Trace

- R1. Ramp entrance on the left side of the playfield, reachable from the left flipper
- R2. Ball physically rolls along guide walls (not teleported) with visible canvas-drawn rails
- R3. Reduced gravity while ball is on the ramp (simulates elevated surface)
- R4. Ball can fail to complete the ramp if launched too slowly — rolls back down
- R5. Ramp exit on the right side, dropping the ball toward the right inlane/flipper area
- R6. Points awarded for completing the ramp (entering the exit zone)
- R7. Ramp visuals drawn with canvas — no image changes needed

## Scope Boundaries

- Canvas-drawn rails only — no changes to table-layer0.png or table-mask.png
- No image layering (ball doesn't go "under" anything visually)
- No ramp-specific sound yet (can reuse sling sound as placeholder)
- Ramp is a single fixed path — no branching, no alternate exits

## Key Technical Decisions

- **Ramp as collision-filtered guide walls:** The ramp is a narrow channel of static wall bodies with a unique collision category. When the ball enters the ramp entrance sensor, its collision mask switches to only collide with ramp walls (not playfield walls). When it exits, the mask switches back. This is the same pattern the v1 game used for its layer system (collision categories with bitmask filtering). Rationale: lets the ball visually cross over playfield elements without getting stuck on them, while still being physically guided by real walls.

- **Reduced gravity via per-frame force:** While the ball is on the ramp, apply an upward force each frame to counteract a portion of gravity (e.g., 60% gravity reduction). This simulates the ball rolling on an elevated, slightly inclined surface. Rationale: simpler than changing engine gravity globally (which affects all bodies), and can be tuned per-ramp.

- **Entrance and exit as sensor zones:** Small sensor bodies at each end of the ramp. Entering the entrance sensor switches the ball to ramp collision mode. Entering the exit sensor switches back to playfield mode, awards points, and gives the ball a velocity boost downward toward the flippers. Rationale: clean state transitions without complex position checking.

- **Ramp path as quadratic bezier:** The ramp follows a smooth curve from lower-left to upper-right. The guide walls are generated by offsetting the bezier centerline by ±half the channel width. This produces two parallel curved walls. Rationale: smooth curves, easy to adjust by changing control points, reuses the curveWall pattern already in the codebase.

- **Approximate coordinates (to be tuned):**
  - Entrance: (~350, 1500) — left side, above the slingshot, reachable from left flipper
  - Curve apex: (~500, 350) — upper-left area, above the three-point arc
  - Exit: (~1100, 600) — right side, upper-mid, drops toward right inlane
  - Channel width: ~70px (ball diameter 48 + clearance)

## Open Questions

### Resolved During Planning

- **Can Matter.js do per-body collision filtering?** Yes — `collisionFilter.category` and `collisionFilter.mask` on each body. The v1 codebase used this exact pattern with 5 collision categories (CAT.ALWAYS, CAT.PLAYFIELD, etc.). The wall-data.json approach creates walls with default category/mask, which can be overridden for ramp walls.

- **Should the ball be visible on the ramp?** Yes — it rolls along the visible canvas-drawn rails. The rails are drawn on top of the table image but under the ball, so the ball appears to ride on them.

### Deferred to Implementation

- **Exact ramp coordinates.** The entrance, apex, and exit positions are approximate. They need play-testing to find the sweet spot where: the left flipper can reach the entrance, the curve feels natural, and the exit feeds the right inlane.

- **Gravity reduction amount.** Start at 60% reduction (ball feels 40% of normal gravity on the ramp). Tune based on whether balls complete the ramp too easily or not at all.

- **Ramp completion scoring value.** Start at 5000 points. Adjust relative to bumper (1000) and slingshot (500) values.

## Implementation Units

- [ ] **Unit 1: Collision filter system**

**Goal:** Add collision categories so ramp walls and playfield walls can be distinguished. Ball's mask switches between playfield mode and ramp mode.

**Requirements:** R3, R4 (enables the ball to travel on ramp without hitting playfield walls)

**Dependencies:** None

**Files:**
- Modify: `v2.html`

**Approach:**
- Define collision categories: PLAYFIELD (default for all existing walls) and RAMP (for ramp guide walls)
- All existing walls get `collisionFilter: { category: PLAYFIELD, mask: BALL }`
- Ball gets `collisionFilter: { category: BALL, mask: PLAYFIELD }` initially
- Add helper functions: `setBallToRamp(ball)` switches mask to RAMP, `setBallToPlayfield(ball)` switches mask to PLAYFIELD
- Existing walls, bumpers, slingshots, and interior walls all use the PLAYFIELD category
- The scoop and its shield wall also use PLAYFIELD

**Patterns to follow:**
- v1 `index.html` had `CAT = { BALL: 0x0001, ALWAYS: 0x0002, PLAYFIELD: 0x0004, LEFT_RAMP: 0x0008, ... }` and `LAYER_MASK` mappings
- Existing `createWall()` function — add optional collision filter parameter

**Test scenarios:**
- Happy path: After adding categories, all existing gameplay is unchanged (balls still collide with all walls, bumpers, slingshots)
- Edge case: Ball collision mask defaults to PLAYFIELD on creation and on serve

**Verification:**
- Play the game with collision filter system added — everything works exactly as before. No visible change. The filter infrastructure is in place but not yet used by any ramp.

- [ ] **Unit 2: Ramp guide walls and canvas rendering**

**Goal:** Create the physical ramp channel (two bezier-curved guide walls) and draw visible rails on the canvas.

**Requirements:** R1, R2, R5, R7

**Dependencies:** Unit 1

**Files:**
- Modify: `v2.html`

**Approach:**
- Define ramp path as a quadratic bezier centerline: entrance point → control point → exit point
- Generate left and right rail walls by offsetting the centerline by ±half channel width, using the same bezier-to-wall-segments pattern as `curveWall()`
- Ramp walls use `collisionFilter: { category: RAMP, mask: BALL }` — only collide with ball when ball is in ramp mode
- Render the rails in the canvas render loop: draw two colored lines (the offset bezier curves) on top of the table image. Use a distinct color that reads as "ramp" (semi-transparent chrome or colored rails)
- Rails drawn BEFORE the ball in render order so the ball appears on top of them

**Patterns to follow:**
- `curveWall()` function generates wall segments from bezier parameters
- `createWall()` function creates static rectangle bodies
- Bumper render loop draws directly in the `render()` function

**Test scenarios:**
- Happy path: Ramp rails are visible on screen as two parallel curved lines from lower-left to upper-right
- Happy path: Shift+D debug shows ramp wall bodies in a distinct color along the rail path
- Edge case: Ramp walls don't interfere with normal gameplay (ball passes through them on the playfield because collision mask doesn't match)

**Verification:**
- Visible rails drawn on the table. Debug overlay shows ramp wall bodies. Ball on the playfield ignores the ramp walls completely.

- [ ] **Unit 3: Entrance and exit sensors with state transitions**

**Goal:** Sensor zones at ramp entrance and exit that switch the ball's collision mask, apply gravity reduction, award score, and handle incomplete ramp attempts.

**Requirements:** R1, R3, R4, R5, R6

**Dependencies:** Unit 2

**Files:**
- Modify: `v2.html`

**Approach:**
- Create entrance sensor (small circle body, `isSensor: true`) at the ramp entrance position
- Create exit sensor at the ramp exit position
- Track ramp state: `onRamp` boolean per ball
- On entrance collision: if ball is moving upward (velocity.y < 0) and fast enough (speed > threshold), switch ball to RAMP mask, set `onRamp = true`
- While `onRamp`: each frame in `update()`, apply upward force to counteract ~60% of gravity
- On exit collision: switch ball back to PLAYFIELD mask, set `onRamp = false`, award RAMP_COMPLETE points, apply downward/rightward velocity boost toward the right inlane, play sound
- Failsafe — ball rolls back: if ball is on ramp but falls below the entrance y-position (rolled back down), switch back to PLAYFIELD mask and set `onRamp = false`. No points awarded.
- The entrance sensor should require the ball to be moving upward to activate — prevents the ball from re-entering the ramp while rolling back down

**Patterns to follow:**
- Scoop sensor: `isSensor: true`, collision detection in the `Events.on('collisionStart')` handler
- Multiball state tracking: `isMultiball` boolean pattern

**Test scenarios:**
- Happy path: Ball enters ramp from left flipper shot → collision mask switches → ball rides along ramp rails → exits on right side → score awarded → ball drops to right inlane
- Happy path: Ramp completion message "RAMP +5,000" flashes on screen
- Edge case: Ball enters ramp too slowly → rolls back down → exits entrance area → collision mask reverts to playfield → no points awarded
- Edge case: Ball on ramp during multiball → only the ball that entered the ramp has its mask changed, other balls remain on playfield
- Edge case: Ball enters launcher re-entry zone while on ramp (shouldn't happen geometrically, but guard against it)
- Integration: Ramp entrance collision triggers mask switch which causes ball to stop colliding with bumpers → ball passes over bumper positions while on ramp rails

**Verification:**
- Complete a ramp shot from the left flipper. Ball visibly travels along the rails, exits on the right, score is awarded. Then intentionally shoot too weakly — ball rolls back, no points, back to normal playfield physics.

## System-Wide Impact

- **Collision filter added to all bodies:** Every existing wall, bumper, slingshot, and interior wall gets a collision filter category. This is invisible but structural — verify nothing breaks.
- **Per-frame gravity override:** The `update()` function gets a new block that applies force to the ball while `onRamp` is true. This interacts with the speed clamp — the upward force may fight the clamp. Ensure the clamp runs after the force application.
- **Multiball interaction:** If multiple balls exist, only the ball that touched the entrance sensor should be in ramp mode. Track which body is on the ramp, not just a boolean.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| Ramp wall positions don't feel right on first try | Coordinates are defined as constants at the top — easy to adjust. Play-test and tune. |
| Ball gets stuck at ramp entrance/exit transition | Entrance and exit sensors are slightly inside the ramp channel, not at the exact edge. Ball has momentum when transitioning. |
| Collision filter change breaks existing gameplay | Unit 1 is pure infrastructure — verify everything still works before adding the ramp. |
| Reduced gravity makes the ball float unnaturally | Tune the gravity reduction factor. Start at 60% reduction, adjust based on feel. |
| Ball on ramp collides with scoop sensor | Scoop sensor uses PLAYFIELD category. Ball on ramp has RAMP mask. They won't collide. |
